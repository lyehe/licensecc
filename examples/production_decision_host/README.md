# Production Decision Host Example

This example shows the production-shaped client path:

- local license verification
- enforced host-integrity policy
- required online verification with primary and backup verifier URLs
- persisted revocation-floor load/store through `lcc_acquire_license_decision()`

Build it with examples enabled:

```console
cmake -S . -B build -DLCC_BUILD_EXAMPLES=ON
cmake --build build --target production_decision_host
```

Run:

```console
production_decision_host ^
  --license path/to/license.lic ^
  --floor-store path/to/revocation-floor.txt ^
  --verifier-url https://primary.example.workers.dev ^
  --verifier-url https://backup.example.workers.dev
```

Verifier URLs must use HTTPS by default. For a local test verifier, pass
`--allow-insecure-http-for-test`. The `--fail-host-integrity-for-test` switch
forces the host-integrity callback to deny so integration tests can verify the
fail-closed path.

The floor store records the maximum accepted `revocation_seq` for the current
project, feature, and license fingerprint. Load or store failures deny the
decision. The example does not print raw hardware identifiers, secrets, or full
license fingerprints.

For verifier deployments that enable request proof-of-possession, wire a
provider into `OnlineClient::request_proof_provider`. The shared online callback
helper builds the canonical payload from the nonce-bound `LccOnlineRequest` so
the host can sign it with a registered device key. Roll this out with the
verifier in `REQUEST_SIGNATURE_MODE=soft` before moving selected entitlements to
`required`.

When this example is built with OpenSSL Crypto, or with the Windows CNG fallback
enabled, it can use a file-backed development key generated by the verifier
package:

```console
npm --prefix services/cloudflare-licensing-backend run device-key -- generate --out-dir .device-key
production_decision_host ^
  --license path/to/license.lic ^
  --floor-store path/to/revocation-floor.txt ^
  --verifier-url https://primary.example.workers.dev ^
  --request-proof-device-key-id sha256:<64 hex key id> ^
  --request-proof-private-key .device-key/device_private_key.pkcs8.pem
```

That file-backed key path is intentionally for integration tests and bootstrap
work. Production hosts should keep the same request-proof callback shape but
store and use the private key through the platform key store or hardware-backed
provider where available.

Host-integrity probes are best-effort signals running on a customer-controlled
machine. Treat them as tamper-resistant telemetry and policy inputs, not proof
that the host is trustworthy.
